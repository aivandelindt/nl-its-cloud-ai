Phase 1 · ~2 weeks
1

Standardize the Workspace

A reproducible toolchain every engineer can spin up in under an hour

Objective

Eliminate "works on my machine" by codifying the entire local dev environment — editor, extensions, CLIs, and a dev container — so onboarding is a single command.

Owners

Platform lead + 1 engineer

Deliverables

devcontainer.json + Dockerfile .vscode/extensions.json SETUP.md

Task checklist

  • Pin VSCode extensions: Copilot, Copilot Chat, Terraform, Azure Tools, Bicep
  • Version-pin OpenTofu and Azure CLI; document supported versions
  • Author devcontainer.json with OpenTofu, Azure CLI, LSP, shell tooling
  • Add extensions.json so the editor prompts on clone
  • Write SETUP.md with one-command bootstrap flow
  • Validate container builds on mac + Windows/WSL

Exit criteria

A new engineer goes from git clone to a working OpenTofu + Azure environment in under 60 minutes, with zero manual tool installs.

Risks

  • Tool version drift across OS — pin everything and test each platform
  • Corporate proxy blocking image pulls — pre-mirror images internally
Phase 2 · ~2 weeks
2

Add MCP Servers

Give the agent eyes and hands: GitHub, Azure, and Terraform over MCP

Objective

Connect Copilot Agent Mode to live tooling through trusted MCP servers, with secrets handled securely, and prove the agent can read real state and act.

Owners

Platform lead + security reviewer

Deliverables

.vscode/mcp.json Secrets runbook MCP allow-list

Task checklist

  • Create .vscode/mcp.json for GitHub, Azure, and Terraform MCP servers
  • Route credentials through input prompts / Key Vault — nothing hardcoded
  • Security review each server before enabling
  • Smoke-test: query subscription, search repo, fetch module doc
  • Guided Agent Mode session: "summarise our dev resource group"
  • Document allow-list and how to request new servers

Exit criteria

In Agent Mode an engineer can ask a natural-language question that triggers a real Azure query and a GitHub action (e.g. open a draft PR), end to end.

Risks

  • Over-permissioned service principals — start read-only, widen deliberately
  • Untrusted third-party servers — only enable reviewed, allow-listed sources
Phase 3 · ~2 weeks
3

Author Instructions

Encode IaC, naming, and tagging standards as always-on rules

Objective

Capture the team's conventions as scoped instruction files so generated code follows your standards automatically, without anyone re-typing them.

Owners

Senior cloud engineers (standards owners)

Deliverables

instructions/terraform.md Naming + tagging standard Before/after evidence

Task checklist

  • Document naming convention <type>-<workload>-<env> and required tags
  • Define region policy, provider-pinning, no-plaintext-secrets rule
  • Write instructions/terraform.md with applyTo **/*.tf
  • Keep rules crisp and testable — short constraints, not essays
  • A/B test: generate same module with and without instructions
  • Iterate until generated HCL passes review with minimal edits

Exit criteria

Code the agent generates conforms to naming, tagging, region, and provider-pinning standards on the first pass, with reviewers making only minor edits.

Risks

  • Instruction bloat diluting model attention — keep them short and rule-like
  • Conflicting or stale rules — assign a single standards owner
Phase 4 · ~3–4 weeks
4

Build Skills + Agent Team

Reusable procedures plus a planner → implementer team

Objective

Move from ad-hoc prompting to repeatable workflows: auto-discovered skills for common procedures and a small team of scoped agents that delegate to each other.

Owners

Platform team (2–3 engineers)

Deliverables

skills/opentofu-review agents/iac-planner.md agents/iac-implementer.md

Task checklist

  • Author skills/opentofu-review/SKILL.md for auto-discovery
  • Build read-only iac-planner (codebase + azure-mcp + terraform-mcp)
  • Build iac-implementer that can edit files and run plan/apply
  • Define handoffs: planner delegates after user approval
  • Least privilege: only implementer gets write/apply tools
  • End-to-end rehearsal: small landing-zone change through full team

Exit criteria

The agent team completes a realistic task — plan a dev landing zone, scaffold the .tf files, and self-review — with a human approving the transition between plan and apply.

Risks

  • Agents exceeding scope — enforce least-privilege tool lists per agent
  • Skill descriptions too vague — write task-matching summaries
Phase 5 · ~2 weeks
5

Add Hooks (Guardrails)

Deterministic scripts that always run — formatting, validation, hard stops

Objective

Layer in non-probabilistic guardrails on lifecycle events so destructive actions are blocked and quality checks run every time — regardless of what the model decides.

Owners

Platform + security

Deliverables

hooks/hooks.json hooks/guard.sh Guardrail docs

Task checklist

  • Define hooks/hooks.json with PreToolUse and PostToolUse handlers
  • Write guard.sh to block tofu apply without an approved plan
  • PostToolUse hook: tofu fmt -recursive after edits
  • Test hard-stop: non-zero exit cancels destructive op
  • Wire tofu validate + policy check into pre-apply gate
  • Document hook behaviour for engineers

Exit criteria

A tofu apply against prod without a reviewed plan is reliably blocked by the hook, and HCL is auto-formatted after every agent edit — with no reliance on the model remembering.

Risks

  • Over-aggressive hooks blocking legitimate work — log clearly, allow override paths
  • Hooks not running on all machines — ship them inside the plugin (Phase 6)
Phase 6 · ~2–3 weeks
6

Package & Publish

Bundle everything into one shippable plugin for the whole team

Objective

Consolidate instructions, skills, agents, MCP servers, and hooks into a single versioned plugin, publish it internally, and drive team-wide adoption.

Owners

Platform lead + enablement

Deliverables

Packaged plugin Install docs Maintenance plan

Task checklist

  • Create plugin.json manifest wiring all six customization layers
  • Bundle .mcp.json so servers and hooks travel with the plugin
  • Version with semver and write a CHANGELOG
  • Publish to internal registry / shared repo with install instructions
  • Run an enablement session and capture a feedback channel
  • Define ownership + update cadence (who maintains, how fixes ship)

Exit criteria

The whole IT Cloud Team installs one plugin and inherits the full governed setup — standards, agents, and guardrails — with a clear owner and update process in place.

Risks

  • Manifest keys / MCP names drifting with vendor updates — verify docs before each release
  • Adoption stalling — pair launch with hands-on enablement, not just a link
