I

The evolution of intelligence

What an LLM actually is

An LLM is a very large neural network trained to predict the next token. Everything it "knows" is compressed into weights learned from text. It is a brilliant pattern engine — not a database, not a reasoner with a will.

Knowledge cutoffFrozen at training time
No memoryBetween calls
No system accessCannot touch your infra
HallucinationConfident but wrong

A raw model is a powerful brain in a jar. Tools, context, and a loop turn it into an agent.

The five-step climb

LLMPredicts text from patterns in weights
+ RAGRetrieves your docs at query time — grounded, current, private knowledge
+ ToolsCalls functions — read a file, run a query, hit an API
+ MCPStandardizes how tools plug in — any client talks to any server
AgentCloses the loop: plan, act, observe, adapt until the goal is met

Copilot: from autocomplete to coworker

Completions

Chat / Edit

Ask questions, refactor across files. The model proposes; you review and apply.

Agent Mode

Give a goal. It plans, edits files, runs terminal commands, calls MCP tools, and iterates until done.

MCP is the USB-C for AI — an open standard (now under the Linux Foundation) that defines a single, common way for AI clients to connect to tools and data sources. Build a connector once — every MCP-aware host can use it.

V

Building the Azure IaC Copilot plugin

Six customization layers bundle into one shippable unit — instructions, prompts, skills, agents, MCP servers, and hooks.

Instructions

Always-on rules & context applied to every task

Prompt files

Reusable, manually-invoked workflows (/deploy, /review)

Skills

Auto-discovered procedures when relevant

Custom agents

Named personas with own model, tools, scope

MCP servers

Live connections to external tools and data

Hooks

Deterministic scripts on lifecycle events — guardrails

Layer 1 — Instructions

instructions/terraform.mdMarkdown
---
applyTo: "**/*.tf"
---
# OpenTofu / Azure standards

- Pin every provider with a ~> constraint.
- Never hardcode secrets — read from
  Azure Key Vault data sources.
- All resources carry owner, environment,
  and managed_by tags.
- Default region is westeurope unless the
  user states otherwise.
- Use remote azurerm backend; never local state.
- Resource names follow:
  <type>-<workload>-<env> (rg-payments-dev).

Instructions apply automatically via applyTo globs. Keep them short and rule-like — crisp, testable constraints.

Layer 2 — Skills

skills/opentofu-review/SKILL.mdMarkdown
---
name: opentofu-review
description: Validate & review an OpenTofu
  module before plan/apply. Use whenever
  .tf files change.
---
# OpenTofu module review

When invoked:
1. Run `tofu fmt -check` and fix drift.
2. Run `tofu validate`.
3. Confirm all providers are version-pinned.
4. Flag any resource missing required tags.
5. Confirm no plaintext secrets exist.
6. Summarise findings as a checklist.

Skills are auto-discovered by the agent (reads description first, loads full skill when matched). Prompt files are manually invoked with a slash command.

Layer 3 — Custom agents

agents/iac-planner.mdMarkdown
---
name: iac-planner
description: Designs Azure architecture and
  produces an OpenTofu plan. Read-only.
tools:
  - codebase
  - azure-mcp
  - terraform-mcp
handoffs:
  - iac-implementer
---
You are a senior cloud architect. Inspect
the current Azure estate and the repo, then
propose a minimal, well-tagged OpenTofu
change set. Do NOT write files — hand off to
iac-implementer once the user approves.

Least privilege: planner is read-only. Only iac-implementer can edit files and run apply. Agents hand off to form a team: architect → builder → reviewer.

Layer 4 — Hooks

hooks/hooks.jsonJSON
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "run_in_terminal",
        "command": "./hooks/guard.sh",
        "description": "Block tofu apply without an approved plan"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "edit_file",
        "command": "tofu fmt -recursive",
        "description": "Auto-format HCL after edits"
      }
    ]
  }
}

Hooks are deterministic — they always run on lifecycle events. A non-zero exit blocks destructive actions.

Security & governance

Trust MCP sources

Only add servers you trust. Review third-party tools like any dependency.

Org policy

Centrally allow/deny MCP servers. Require approval for new tools.

Secrets via Key Vault

No credentials in code. Input prompts, Key Vault, managed identities.

Human in the loop

Require approval for terminal commands and destructive ops. Hooks enforce it deterministically.

VII

Adoption roadmap

Six sequential phases from a standardized workspace to a shipped, governed Copilot plugin. Estimated end-to-end timeline: 13–17 weeks with a small platform team.

P1 P2 P3 P4 P5 P6

Objective: Eliminate "works on my machine" by codifying the entire local dev environment — editor, extensions, CLIs, and a dev container — so onboarding is a single command.

Task checklist
  • Pin the VSCode extension set: GitHub Copilot, Copilot Chat (Agent Mode), HashiCorp Terraform, Azure Tools, Bicep.
  • Install and version-pin OpenTofu and Azure CLI; document supported versions.
  • Author a devcontainer.json bundling OpenTofu, Azure CLI, tofu/terraform LSP, and shell tooling.
  • Add .vscode/extensions.json with recommended extensions so the editor prompts on clone.
  • Write a CONTRIBUTING / SETUP.md with the one-command bootstrap flow.
  • Validate the container builds clean on a fresh machine (mac + Windows/WSL).
Deliverables
devcontainer.json + Dockerfile.vscode/extensions.jsonSETUP.md
Suggested owners

Platform lead + 1 engineer

Exit criteria
A new engineer goes from git clone to a working OpenTofu + Azure environment in under 60 minutes, with zero manual tool installs.
Key risks & mitigations
  • Tool version drift across OS — pin everything and test on each platform.
  • Corporate proxy/firewall blocking container image pulls — pre-mirror images internally.

Objective: Connect Copilot Agent Mode to live tooling through trusted MCP servers, with secrets handled securely, and prove the agent can read real state and act.

Task checklist
  • Create .vscode/mcp.json defining the GitHub, Azure, and Terraform MCP servers.
  • Route all credentials through input prompts / Key Vault — nothing hardcoded.
  • Have security review each server before enabling (treat like a dependency).
  • Smoke-test each server in isolation: query a subscription, search a repo, fetch a module doc.
  • Run a guided Agent Mode session: "summarise our dev resource group" to confirm tool calls work.
  • Document the allow-list and how engineers request a new server.
Deliverables
.vscode/mcp.jsonSecrets runbookMCP allow-list
Suggested owners

Platform lead + security reviewer

Exit criteria
In Agent Mode an engineer can ask a natural-language question that triggers a real Azure query and a GitHub action (e.g. open a draft PR), end to end.
Key risks & mitigations
  • Over-permissioned service principals — start read-only, widen deliberately.
  • Untrusted third-party servers — only enable reviewed, allow-listed sources.

Objective: Capture the team's conventions as scoped instruction files so generated code follows your standards automatically, without anyone re-typing them.

Task checklist
  • Document current naming convention (<type>-<workload>-<env>) and required tags.
  • Define region policy, provider-pinning rules, and the no-plaintext-secrets rule.
  • Write instructions/terraform.md with an applyTo glob scoped to **/*.tf.
  • Keep rules crisp and testable — short constraints, not essays.
  • A/B test: generate the same module with and without instructions; compare drift.
  • Iterate until generated HCL passes review with minimal manual correction.
Deliverables
instructions/terraform.mdNaming + tagging standardBefore/after evidence
Suggested owners

Senior cloud engineers (standards owners)

Exit criteria
Code the agent generates conforms to naming, tagging, region, and provider-pinning standards on the first pass, with reviewers making only minor edits.
Key risks & mitigations
  • Instruction bloat diluting model attention — keep them short and rule-like.
  • Conflicting or stale rules — assign a single standards owner.

Objective: Move from ad-hoc prompting to repeatable workflows: auto-discovered skills for common procedures and a small team of scoped agents that delegate to each other.

Task checklist
  • Author skills/opentofu-review/SKILL.md with a clear description for auto-discovery.
  • Build a read-only iac-planner agent (codebase + azure-mcp + terraform-mcp).
  • Build an iac-implementer agent that can edit files and run plan/apply.
  • Define handoffs so the planner delegates to the implementer after approval.
  • Apply least privilege: only the implementer gets write/apply tools.
  • End-to-end rehearsal: drive a small landing-zone change through the full team.
Deliverables
skills/opentofu-reviewagents/iac-planner.mdagents/iac-implementer.md
Suggested owners

Platform team (2–3 engineers)

Exit criteria
The agent team completes a realistic task — plan a dev landing zone, scaffold the .tf files, and self-review — with a human approving the transition between plan and apply.
Key risks & mitigations
  • Agents exceeding scope — enforce least-privilege tool lists per agent.
  • Skill descriptions too vague to be discovered — write task-matching summaries.

Objective: Layer in non-probabilistic guardrails on lifecycle events so destructive actions are blocked and quality checks run every time — regardless of what the model decides.

Task checklist
  • Define hooks/hooks.json with PreToolUse and PostToolUse handlers.
  • Write guard.sh to block tofu apply without an approved, reviewed plan.
  • Add a PostToolUse hook to run tofu fmt -recursive after edits.
  • Test the hard-stop path: confirm a non-zero exit cancels a destructive op.
  • Wire validation (tofu validate + policy check) into the pre-apply gate.
  • Document hook behaviour so engineers understand why an action was blocked.
Deliverables
hooks/hooks.jsonhooks/guard.shGuardrail docs
Suggested owners

Platform + security

Exit criteria
A tofu apply against prod without a reviewed plan is reliably blocked by the hook, and HCL is auto-formatted after every agent edit — with no reliance on the model remembering.
Key risks & mitigations
  • Over-aggressive hooks blocking legitimate work — log clearly and allow override paths.
  • Hooks not running on all machines — ship them inside the plugin (Phase 6).

Objective: Consolidate instructions, skills, agents, MCP servers, and hooks into a single versioned plugin, publish it internally, and drive team-wide adoption.

Task checklist
  • Create plugin.json manifest wiring up all six customization layers.
  • Bundle .mcp.json so servers and hooks travel with the plugin.
  • Version with semver and write a CHANGELOG.
  • Publish to your internal registry / shared repo with install instructions.
  • Run an enablement session and capture a feedback channel.
  • Define an ownership + update cadence (who maintains, how fixes ship).
Deliverables
Packaged pluginInstall docsMaintenance plan
Suggested owners

Platform lead + enablement

Exit criteria
The whole IT Cloud Team installs one plugin and inherits the full governed setup — standards, agents, and guardrails — with a clear owner and update process in place.
Key risks & mitigations
  • Manifest keys / MCP names drifting with vendor updates — verify against current docs before each release.
  • Adoption stalling — pair the launch with hands-on enablement, not just a link.

Timelines assume a 2–3 person platform team running phases roughly sequentially with some overlap. Verify exact plugin manifest keys, MCP server package names, and hook event names against current GitHub Copilot & MCP documentation before each release.
