What an LLM actually is

The core idea

An LLM is a very large neural network trained to predict the next token. Everything it "knows" is compressed into weights learned from text. It is a brilliant pattern engine — not a database, not a reasoner with a will.

Built-in limits

  • Frozen at a knowledge cutoff
  • No memory between calls
  • Cannot touch your systems
  • Can hallucinate confidently

Takeaway: a raw model is a powerful brain in a jar. The rest of this deck is about giving it eyes, hands, and judgment.

MCP

Model Context Protocol — the USB-C for AI

MCP is an open standard (now under the Linux Foundation) that defines a single, common way for AI clients to connect to tools and data sources. Build a connector once — every MCP-aware host can use it.

Before MCP

N clients × M tools = N×M bespoke integrations. Brittle and unscalable.

With MCP

One protocol. N + M connectors. Any client, any server, plug-and-play.

Under the hood

JSON-RPC 2.0 messages over stdio (local) or HTTP/SSE (remote).

MCP

MCP architecture & primitives

The three roles

  • Host — the app (VSCode + Copilot) the user works in
  • Client — lives in the host, one per server connection
  • Server — exposes capabilities (GitHub, Azure, Terraform)

What servers expose

  • Tools — actions the model can invoke
  • Resources — data/files it can read
  • Prompts — reusable templated workflows

      "command": "docker",

    }
  },
  "inputs": [
    { "id": "gh_pat", "type": "promptString",
      "description": "GitHub PAT", "password": true }
  ]
}

Azure MCP — query subscriptions, resources, AKS, KQL

Terraform MCP — registry, modules, provider docs

Secrets go through inputs, never hardcoded. VSCode prompts once and stores them securely.

Build a Plugin

Six layers of customization

Instructions

Always-on rules & context the model applies to every task.

Prompt files

Reusable, manually-invoked workflows (/deploy, /review).

Skills

Auto-discovered procedures the agent pulls in when relevant.

Custom agents

Named personas with their own model, tools, and scope.

MCP servers

Live connections to external tools and data.

Hooks

Deterministic scripts on lifecycle events — guardrails.

Build a Plugin

Layer 1 — Instructions

instructions/terraform.mdMD
---
applyTo: "**/*.tf"
---
# OpenTofu / Azure standards

- Pin every provider with a ~> constraint.
- Never hardcode secrets — read from
  Azure Key Vault data sources.
- All resources carry owner, environment,
  and managed_by tags.
- Default region is westeurope unless the
  user states otherwise.
- Use remote azurerm backend; never local
  state.
- Resource names follow:
  <type>-<workload>-<env> (rg-payments-dev).

Why this matters

Instructions are applied automatically to matching files via applyTo globs. The agent inherits your team's conventions without anyone re-typing them.

Scope wisely

Keep them short and rule-like. Long essays dilute attention — prefer crisp, testable constraints.

Layer 2 — Skills

skills/opentofu-review/SKILL.mdMD
---
name: opentofu-review
description: Validate & review an OpenTofu
  module before plan/apply. Use whenever
  .tf files change.
---
# OpenTofu module review

When invoked:
1. Run `tofu fmt -check` and fix drift.
2. Run `tofu validate`.
3. Confirm all providers are version-pinned.
4. Flag any resource missing required tags.
5. Confirm no plaintext secrets exist.
6. Summarise findings as a checklist.

Progressive disclosure

The agent reads only the description until a task matches — then loads the full skill. Cheap to have many.

Skills vs prompts

Skills are auto-discovered by the agent; prompt files are manually invoked by the user with a slash command.

Layer 3 — Custom agents

agents/iac-planner.mdMD
---
name: iac-planner
description: Designs Azure architecture and
  produces an OpenTofu plan. Read-only.
tools:
  - codebase
  - azure-mcp
  - terraform-mcp
handoffs:
  - iac-implementer
---
You are a senior cloud architect. Inspect
the current Azure estate and the repo, then
propose a minimal, well-tagged OpenTofu
change set. Do NOT write files — hand off to
iac-implementer once the user approves.

Least privilege

The planner gets read-only tools. Only the iac-implementer can edit files and run apply.

Handoffs

Agents delegate to one another, forming a small team: architect → builder → reviewer. Each has a focused persona and toolset.

Layer 4 — Hooks (guardrails)

hooks/hooks.jsonJSON
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "run_in_terminal",
        "command": "./hooks/guard.sh",
        "description": "Block tofu apply without an approved plan"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "edit_file",
        "command": "tofu fmt -recursive",
        "description": "Auto-format HCL after edits"
      }
    ]
  }
}

Deterministic, not probabilistic

Hooks are plain scripts on lifecycle events (PreToolUse, PostToolUse, SessionStart…). They always run — unlike a model that might forget.

Hard stops

A non-zero exit can block a destructive action — e.g. refuse tofu apply in prod without a reviewed plan.

Security & governance

Trust your MCP sources

Only add servers you trust. Treat third-party tools like any dependency — review before enabling.

Org policy controls

Enterprises can centrally allow/deny MCP servers and require approval for new tools.

Secrets via Key Vault

No credentials in code or config. Use input prompts, Key Vault, and managed identities.

Human in the loop

Require approval for terminal commands and destructive ops. Hooks enforce it deterministically.
